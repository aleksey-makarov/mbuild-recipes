# Анализ этапов bootstrap в Linux From Scratch 12.4-systemd

## Общая схема bootstrap

LFS строит полноценную Linux-систему в три стадии, каждая из которых решает конкретную проблему зависимости от хост-системы:

| Стадия | Главы | Суть | Где запускаются инструменты | Куда ставятся результаты |
|--------|-------|------|---------------------------|-------------------------|
| **1. Кросс-тулчейн** | 5 | Собрать компилятор, который **работает на хосте**, но **генерирует код для LFS** | На хосте | `$LFS/tools/` |
| **2. Временные утилиты** | 6 | Кросс-компилировать базовые утилиты целевой системы | На хосте (кросс-компилятором) | `$LFS/usr/` (через DESTDIR) |
| **3. Chroot + финальная сборка** | 7–8 | Войти в chroot и пересобрать всё нативно, уже без зависимости от хоста | В chroot (`$LFS` = `/`) | `/usr/` |

### Ключевая идея: "Фальшивая" кросс-компиляция

LFS собирает систему для **той же** архитектуры, на которой работает хост. Но используется техника кросс-компиляции — переменная `$LFS_TGT` (например, `x86_64-lfs-linux-gnu`) специально отличается от хостового триплета (`x86_64-pc-linux-gnu`), чтобы тулчейн **не мог случайно использовать библиотеки хоста**. Изменён только vendor-поле (`lfs` вместо `pc`).

### Два ключевых каталога до chroot

- **`$LFS/tools/`** — сюда ставится сам кросс-тулчейн (binutils, gcc). Это инструменты, которые **запускаются на хосте**, но генерируют код для целевой системы. Этот каталог **не войдёт** в финальную систему.
- **`$LFS/usr/`** (и `$LFS/usr/lib/`, `$LFS/usr/bin/` и т.д.) — сюда ставятся библиотеки и утилиты **целевой** системы. Они линкуются с glibc целевой системы и предназначены для запуска внутри chroot/финальной системы.

### Опция `--with-sysroot=$LFS`

Кросс-линкер и кросс-компилятор собираются с `--with-sysroot=$LFS`. Это значит, что когда они ищут заголовки и библиотеки, они ищут их относительно `$LFS`, а не `/`. Например, `$LFS/usr/include` вместо `/usr/include`.

## Глава 5: Компиляция кросс-тулчейна

Все пакеты в этой главе **собираются хостовым компилятором** и **устанавливаются в `$LFS/tools/`** (кроме glibc и Linux headers, которые идут в `$LFS/usr/`). Они работают на хосте, но нацелены на целевую систему.

### 5.2. Binutils-2.45 — Pass 1

**Что:** Ассемблер (`as`) и линкер (`ld`) — первый пакет, потому что configure-скрипты GCC и Glibc тестируют ассемблер и линкер для определения доступных фич.

**Configure:**
```
--prefix=$LFS/tools    # ставить в $LFS/tools/
--with-sysroot=$LFS    # искать библиотеки относительно $LFS
--target=$LFS_TGT      # генерировать код для целевого триплета
```

**Линковка:** Статическая + динамическая с библиотеками хоста (это инструменты, которые запускаются на хосте).

**Куда устанавливается:** `$LFS/tools/bin/` и `$LFS/tools/$LFS_TGT/bin/` (hard-linked между собой). В финальной системе **не используется** — это только инструмент сборки.

**Важно для bootstrap:** Binutils ставится первым, потому что configure-скрипты GCC и Glibc выполняют feature-тесты ассемблера и линкера. Неправильно сконфигурированный линкер → сломанный тулчейн, который может проявиться только в конце сборки.

### 5.3. GCC-15.2.0 — Pass 1

**Что:** Кросс-компилятор C/C++. Минимальная сборка — только языки C и C++, отключены почти все runtime-библиотеки.

**Configure:**
```
--target=$LFS_TGT         # генерировать код для LFS
--prefix=$LFS/tools       # поставить в tools
--with-sysroot=$LFS       # sysroot = $LFS
--with-newlib              # нет работающей libc → define inhibit_libc
--without-headers          # не искать заголовки целевой системы
--disable-shared           # только статическая линковка внутренних библиотек
--disable-threads          # нет threading (нет libc)
--disable-libstdcxx        # нет C++ stdlib (зависит от glibc, которой ещё нет)
--enable-languages=c,c++
```

**Линковка:** Сам компилятор линкуется с библиотеками хоста и запускается на хосте. Но код, который он **генерирует**, предназначен для целевой системы.

**Куда:** `$LFS/tools/bin/$LFS_TGT-gcc`, `$LFS/tools/bin/$LFS_TGT-g++`. В финальной системе **не используется**.

**Почему `--with-newlib` и `--without-headers`:** На этом этапе glibc ещё не собрана для целевой системы. GCC нужна libc для сборки своих runtime-библиотек, но мы её ещё не имеем. `--with-newlib` определяет константу `inhibit_libc`, которая отключает код, зависящий от libc. Собирается только `libgcc` — минимальная runtime-библиотека компилятора.

**Дополнительный шаг:** После установки создаётся полный `limits.h`, который включает системный `limits.h`. На момент сборки GCC Pass 1 `$LFS/usr/include/limits.h` ещё не существует, поэтому headers нужно дополнить вручную.

### 5.4. Linux-6.16.1 API Headers

**Что:** Заголовочные файлы ядра Linux, определяющие интерфейс системных вызовов (ABI ядра).

**Линковка:** Никакой — это только `.h` файлы. `make headers` + копирование.

**Куда:** `$LFS/usr/include/`. Это уже каталог **целевой системы** (не tools!). Glibc, которая будет собрана следующей, найдёт их через sysroot.

**Важно для bootstrap:** Glibc компилируется под конкретное ядро. Заголовки ядра определяют, какие системные вызовы доступны. Должны быть установлены **до** сборки glibc.

### 5.5. Glibc-2.42

**Что:** Основная C-библиотека целевой системы. Содержит libc.so, libm.so, ld-linux (динамический линкер) и другие.

**Configure:**
```
--host=$LFS_TGT              # кросс-компиляция: использовать $LFS_TGT-gcc
--build=$(../scripts/config.guess)  # включить режим кросс-компиляции
--prefix=/usr                 # prefix = /usr (относительно sysroot)
--with-headers=$LFS/usr/include  # где искать заголовки ядра
```

**Сборка:**
```
make DESTDIR=$LFS install     # установить с DESTDIR для попадания в $LFS/usr/
```

**Линковка:** Собирается кросс-компилятором (`$LFS_TGT-gcc`). Glibc очень самодостаточна в своей сборочной системе — она практически не полагается на настройки тулчейна по умолчанию, явно использует `CC="$LFS_TGT-gcc"` и флаги `-nostdinc`, `-isystem`.

**Куда:** `$LFS/usr/lib/libc.so.6`, `$LFS/usr/lib/ld-linux-x86-64.so.2` и т.д. Это **финальное расположение** (после chroot `$LFS` станет `/`, и эти файлы окажутся в `/usr/lib/`). Но на стадии 3 glibc будет **пересобрана** начисто.

**Важно для bootstrap:** Это первый момент, когда появляется libc для целевой системы. Без неё невозможно собрать ни одну полноценную программу для целевой платформы. Порядок `Binutils → GCC → Linux Headers → Glibc` критичен: каждый следующий пакет зависит от предыдущих.

**Проверка:** После установки проверяется, что кросс-компилятор корректно линкует с новой glibc:
```
echo 'int main(){}' | $LFS_TGT-gcc -xc -
readelf -l a.out | grep ld-linux
# Должен показать: [Requesting program interpreter: /lib64/ld-linux-x86-64.so.2]
```

### 5.6. Libstdc++ из GCC-15.2.0

**Что:** Стандартная библиотека C++. Была отложена при GCC Pass 1, потому что зависит от glibc.

**Configure:**
```
--host=$LFS_TGT                    # кросс-компиляция
--build=$(../config.guess)
--prefix=/usr                       # целевой prefix
--disable-nls
--disable-libstdcxx-pch
--with-gxx-include-dir=/tools/$LFS_TGT/include/c++/15.2.0
```

**Сборка:** `make DESTDIR=$LFS install`

**Линковка:** Собирается кросс-компилятором. Линкуется с glibc целевой системы (той, что только что установлена в `$LFS/usr/lib/`).

**Куда:** `$LFS/usr/lib/libstdc++.so.*`. Заголовки — в `$LFS/tools/$LFS_TGT/include/c++/` (чтобы кросс-компилятор их находил через sysroot).

## Глава 6: Кросс-компиляция временных утилит

### Общее для всех пакетов главы 6

Все пакеты в этой главе:

- **Компилируются кросс-компилятором** из главы 5 (`$LFS_TGT-gcc`)
- **Configure использует:**
  ```
  --prefix=/usr
  --host=$LFS_TGT
  --build=$(build-aux/config.guess)
  ```
- **Устанавливаются через:** `make DESTDIR=$LFS install`
- **Линкуются динамически** с glibc целевой системы (`$LFS/usr/lib/libc.so.6`)
- **Физически попадают в:** `$LFS/usr/bin/`, `$LFS/usr/lib/` и т.д.
- **В финальной системе:** После chroot это будет `/usr/bin/`, `/usr/lib/`. Но все эти пакеты будут **пересобраны** в главе 8, так что это временные версии.

`--host=$LFS_TGT` заставляет систему сборки использовать кросс-компилятор (ищет `$LFS_TGT-gcc`), а `--build=...` отличающийся от `--host` включает режим кросс-компиляции.

`DESTDIR=$LFS` перенаправляет `make install`, чтобы файлы попадали не в `/usr/` хостовой системы, а в `$LFS/usr/`.

Эти программы **не могут быть запущены на хосте** (скомпилированы для другого sysroot/libc), но и на целевой системе пока не запускаются — chroot ещё не выполнен.

### Binutils-2.45 — Pass 2

**Зачем второй проход:** Нужен **нативный** линкер и ассемблер для целевой системы (тот, что будет работать в chroot).

**Configure:**
```
--prefix=/usr
--host=$LFS_TGT        # кросс-компиляция
--build=$(../config.guess)
--enable-shared
--disable-nls
--enable-64-bit-bfd
--enable-new-dtags
--enable-default-hash-style=gnu
```

**Сборка:** `make DESTDIR=$LFS install`

**Отличие от Pass 1:** Pass 1 создал кросс-инструменты (работают на хосте, нацелены на LFS). Pass 2 создаёт **нативные** инструменты (работают в LFS, нацелены на LFS). Pass 2 собирается **кросс-компилятором** для целевой платформы.

**Куда:** `$LFS/usr/bin/as`, `$LFS/usr/bin/ld` (→ в chroot: `/usr/bin/as`, `/usr/bin/ld`).

### GCC-15.2.0 — Pass 2

**Зачем:** Нативный компилятор, который будет работать **внутри** chroot-среды.

**Configure:**
```
--host=$LFS_TGT
--build=$(../config.guess)
--prefix=/usr
--with-build-sysroot=$LFS
--enable-default-pie
--enable-default-ssp
--disable-nls
--disable-multilib
--disable-libatomic
--disable-libgomp
--disable-libquadmath
--disable-libssp
--disable-libvtv
--enable-languages=c,c++
```

**Сборка:** `make DESTDIR=$LFS install`

**Отличие от Pass 1:** Это уже нативный компилятор для целевой системы (не кросс). Он сам будет запускаться в chroot и компилировать всё остальное. Многие runtime-библиотеки всё ещё отключены — они будут собраны в финальном GCC (глава 8).

**Куда:** `$LFS/usr/bin/gcc`, `$LFS/usr/bin/g++` (→ в chroot: `/usr/bin/gcc`).

**Критический момент bootstrap:** После этого шага у целевой системы есть собственный нативный компилятор. Кросс-компиляция завершена. Известно, что GCC Pass 2 «ломает» кросс-тулчейн — это не считается проблемой, потому что кросс-компиляция больше не нужна.

**Создание симлинка:** `ln -sv gcc $LFS/usr/bin/cc` — многие пакеты ожидают компилятор по имени `cc`.

## Глава 7: Вход в chroot и дополнительные временные инструменты

### Подготовка к chroot

Перед входом в chroot:

1. **Смена владельца:** `chown -R root:root $LFS/{usr,lib,var,etc,bin,sbin,tools,lib64}` — файлы принадлежали пользователю `lfs`, теперь нужен root.
2. **Монтирование виртуальных ФС:** `/dev`, `/proc`, `/sys`, `/run` монтируются (bind-mount и tmpfs) внутри `$LFS`.
3. **Вход в chroot:**
   ```
   chroot "$LFS" /usr/bin/env -i \
       HOME=/root TERM="$TERM" \
       PS1='(lfs chroot) \u:\w\$ ' \
       PATH=/usr/bin:/usr/sbin \
       MAKEFLAGS="-j$(nproc)" \
       TESTSUITEFLAGS="-j$(nproc)" \
       /bin/bash --login
   ```

**Критический момент:** После chroot `$LFS` становится `/`. Все пакеты из глав 5–6, установленные в `$LFS/usr/`, теперь доступны как `/usr/`. Кросс-тулчейн в `$LFS/tools/` доступен как `/tools/` — но больше не нужен.

`PATH=/usr/bin:/usr/sbin` — `/tools/` намеренно **не** включён в PATH. Используются только нативные инструменты из `/usr/bin/`.

### Общее для пакетов главы 7

Все пакеты в этой главе:

- **Компилируются нативным GCC** из Pass 2 (теперь это просто `/usr/bin/gcc`)
- **Configure:**
  ```
  --prefix=/usr
  ```
  (обычный нативный configure, без --host, без --build, без DESTDIR)
- **Линкуются динамически** с `/usr/lib/libc.so.6` (glibc целевой системы)
- **Устанавливаются напрямую** в `/usr/` (мы уже в chroot)
- **Запускаются** прямо здесь, в chroot
- **Будут пересобраны** в главе 8

Эти пакеты нужны как **build-зависимости** для пакетов главы 8, но не были собраны раньше, потому что для кросс-компиляции они были бы слишком сложны или имели бы проблемы с feature detection.

После сборки пакетов главы 7:

- Удаляется документация и статические библиотеки из временных пакетов

## Глава 8: Установка финальной системы

### Общее для всех пакетов главы 8

Все пакеты в этой главе:

- **Компилируются нативным GCC** в chroot-окружении
- **Configure:** `--prefix=/usr` (стандартная нативная сборка)
- **Линкуются динамически** с `/usr/lib/libc.so.6`
- **Устанавливаются напрямую** в `/usr/` (и другие каталоги: `/etc/`, `/var/` и т.д.)
- **Это финальные версии**, которые останутся в готовой системе
- Многие пакеты **пересобираются** заново, даже если уже были установлены. Причины: (а) при кросс-компиляции могли быть пропущены optional features, (б) autoconf мог неправильно определить фичи, (в) нужна стабильность (повторная установка на готовой системе должна дать тот же результат)

### Пакеты (в порядке установки)

#### Базовые данные и документация

| # | Пакет | Заметки |
|---|-------|---------|
| 1 | **Man-pages-6.15** | Только данные (man-страницы), без компиляции |
| 2 | **Iana-Etc-20250807** | `/etc/services` и `/etc/protocols` — данные, без компиляции |

#### Glibc-2.42 — финальная сборка

**Это ключевой момент всей главы 8.** Glibc пересобирается первой, потому что от неё зависит абсолютно всё.

**Configure:**
```
--prefix=/usr
--disable-werror
--enable-kernel=4.19      # минимальная версия ядра
--enable-stack-protector=strong
--disable-nscd
libc_cv_slibdir=/usr/lib
```

**Устанавливается в:** `/usr/lib/libc.so.6`, `/usr/lib/ld-linux-x86-64.so.2`, `/usr/lib/libm.so.6`, `/usr/lib/libpthread.so.0` и т.д.

**После установки:** настраивается `/etc/nsswitch.conf`, часовые пояса, locale.

**Важно:** Это финальная glibc. Все последующие пакеты будут линковаться с ней.

## Резюме: Как работает bootstrap

### Проблема курицы и яйца

Чтобы собрать Linux-систему, нужен компилятор. Чтобы собрать компилятор, нужна libc. Чтобы собрать libc, нужен компилятор. LFS разрывает этот цикл через три прохода:

### Схема разрешения циклических зависимостей

```
Хостовый GCC
    │
    ├─► Binutils Pass 1 (кросс-ассемблер/линкер)
    │       ↓
    ├─► GCC Pass 1 (кросс-компилятор, без libc, без libstdc++)
    │       ↓
    ├─► Linux Headers (только .h файлы)
    │       ↓
    ├─► Glibc (кросс-компилированная для целевой системы)
    │       ↓
    ├─► Libstdc++ (теперь можно, есть glibc)
    │       ↓
    ├─► Временные утилиты (кросс-скомпилированные)
    │       ↓
    ├─► Binutils Pass 2 (нативный для целевой)
    │       ↓
    └─► GCC Pass 2 (нативный для целевой)
            ↓
        ═══ CHROOT ═══
            ↓
        Доп. инструменты (Gettext, Bison, Perl, Python...)
            ↓
        Финальная Glibc
            ↓
        Финальный GCC
            ↓
        Вся остальная система
```

### Три «поколения» тулчейна

1. **Хостовый тулчейн** → собирает кросс-тулчейн (поколение 0 → поколение 1)
2. **Кросс-тулчейн** → собирает нативный тулчейн + утилиты (поколение 1 → поколение 2)
3. **Нативный тулчейн (в chroot)** → пересобирает всё финально (поколение 2 → поколение 3)

К моменту финальной сборки в главе 8 система полностью изолирована от хоста. Финальный GCC собран финальным GCC (предыдущего поколения) с финальной Glibc. Все зависимости замкнуты на саму себя.

### Почему пересобирать в главе 8?

- Кросс-скомпилированные пакеты могут иметь неправильно определённые optional features (autoconf не может выполнить feature tests при кросс-компиляции)
- Нужна гарантия стабильности: повторная установка пакета на готовой системе должна давать тот же результат
- Можно запускать полные тестовые наборы
